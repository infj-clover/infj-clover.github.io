## 개요

데이터 엔지니어링의 핵심은 **데이터를 효율적으로 이동(ETL)시키고 목적에 맞게 가공하는 것**이다. 소스(Source) 시스템과 목적지(Sink)의 성격에 따라 데이터 모델링과 파이프라인 설계 전략은 완전히 달라진다.

데이터 아키텍처의 근간이 되는 **OLTP**와 **OLAP**의 기술적 차이를 정리한다.

---

## 1. 개념 정의

### OLTP (Online Transactional Processing)
- **목적:** 실시간 서비스 운영 (결제, 주문, 회원 정보 관리 등)
- **핵심:** 고빈도 저용량 데이터 처리. 데이터 무결성과 낮은 지연 시간(Low Latency)이 최우선이다.
- **주요 작업:** 개별 레코드 단위의 CRUD(INSERT, UPDATE, DELETE).

### OLAP (Online Analytical Processing)
- **목적:** 의사결정을 위한 데이터 분석 (매출 지표 분석, 사용자 행동 패턴 추출 등)
- **핵심:** 저빈도 대용량 데이터 조회. 대량의 데이터를 처리하는 처리량(Throughput)과 복잡한 쿼리 성능이 최우선이다.
- **주요 작업:** 대규모 집계 및 조회를 수행하는 SELECT.

---

## 2. 기술적 특성 비교

| 항목 | OLTP (Transaction) | OLAP (Analytics) |
|------|-------------------|------------------|
| **데이터 범위** | 현재 시점의 최신 데이터 | 과거부터 축적된 이력 데이터 |
| **저장 방식** | 행 기반 (Row-oriented) | 열 기반 (Columnar) |
| **모델링** | 3차 정규화 (중복 제거) | 비정규화 (Join 비용 최적화) |
| **주요 병목** | Lock 경합, 인덱스 오버헤드 | I/O 대역폭, CPU 연산량 |
| **대표 도구** | MySQL, PostgreSQL, Oracle | BigQuery, Snowflake, Redshift |

---

## 3. 데이터 엔지니어링 관점의 설계 전략

데이터 엔지니어는 이 두 환경 사이의 기술적 간극을 메우는 인프라를 설계한다.

1.  **데이터 추출(Extraction):** 운영 DB에 부하를 주지 않기 위해 Read-Replica를 사용하거나, 로그 기반의 CDC(Change Data Capture)를 적용해 변경 데이터를 실시간으로 수집한다.
2.  **데이터 적재(Loading):** 대규모 조회가 원활하도록 데이터 웨어하우스의 특성에 맞춰 파티셔닝(Partitioning)과 클러스터링(Clustering)을 최적화한다.

> **Key Point:** OLTP는 **정확한 기록**에, OLAP은 **빠른 읽기**에 최적화된 구조를 가진다.

---

## 4. 열 기반(Columnar) 저장 방식의 이점

분석 쿼리는 대개 특정 컬럼의 집계(`SUM`, `AVG` 등)를 목적으로 한다.

- **I/O 효율성:** 행 기반 저장소와 달리 필요한 컬럼만 디스크에서 읽어오므로 불필요한 데이터 로드가 없다.
- **압축률:** 동일한 데이터 타입이 연속적으로 저장되므로 압축 효율이 극대화된다. 이는 저장 공간 절약과 더불어 I/O 성능 향상으로 직결된다.

---

## 결론

비즈니스 요구사항에 따라 적합한 아키텍처를 선택하는 것은 데이터 엔지니어의 필수 역량이다. 운영 환경(OLTP)의 트랜잭션 데이터를 통합하여 분석 환경(OLAP)에서 가치를 창출할 수 있는 구조를 설계하는 것이 파이프라인 구축의 본질적인 목표다.
