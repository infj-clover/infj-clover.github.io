## 개요

데이터 엔지니어링의 본질은 결국 복잡한 데이터 흐름을 얼마나 안정적으로 관리하느냐에 달려 있는 것 같다. 현대 비즈니스 환경에서 데이터의 양과 종류가 급격히 늘어나다 보니, 기존의 단순한 Cron 기반 스케줄링보다는 더 체계적인 파이프라인 자동화가 트렌드가 된 느낌이다. 

워크플로우 오케스트레이션 도구인 **Apache Airflow 3.0**은 이러한 요구에 맞춰 파이프라인을 코드로 정의하고 예약하며 모니터링할 수 있는 환경을 제공한다. 이번 포스팅에서는 3.0 버전에서 강조되는 아키텍처와 TaskFlow API를 중심으로, 직접 실습하며 느낀 직관적인 변화들을 짚어보는 것이 좋겠다고 생각한다.

---

## 1. Airflow를 지탱하는 기본 개념들

Airflow를 이해하기 위해서는 우선 **DAG, Task, Operator**라는 세 가지 축을 살펴보는 것이 순서인 것 같다.

- **DAG (Directed Acyclic Graph):** 방향성이 있고 순환하지 않는 그래프 구조로, 전체적인 작업 흐름을 파이썬 코드로 표현하는 설계도라고 볼 수 있다.
- **Task:** DAG 안에서 실행되는 가장 작은 작업 단위다. 각 Task는 독립적으로 돌아가며 자신만의 상태와 로그를 남기는 것이 특징이다.
- **Operator:** 구체적으로 어떤 작업을 할지 정의해둔 템플릿 클래스인데, `PythonOperator`나 `BashOperator` 등이 대표적이라 할 수 있다.

---

## 2. Airflow 3.0 아키텍처: 유연한 역할 분리

Airflow 3.0의 가장 큰 특징은 시스템의 확장성을 위해 **Control Plane**과 **Execution Plane**을 명확히 나누려 했다는 점인 것 같다.

| 계층 (Plane) | 주요 컴포넌트 | 역할에 대한 생각 |
|------|-------------------|------------------|
| **Control Plane** | Scheduler, API Server, Dag Processor | 무엇을 언제 실행할지 판단하고 전체적인 상태를 관리하는 컨트롤 타워 역할이다. |
| **Execution Plane** | Executor, Worker | 판단이 내려진 작업을 실제 환경에서 실행하는 실행부라 할 수 있다. |
| **Metadata DB** | Postgres, MySQL 등 | 실행 이력이나 상태 정보 등을 차곡차곡 쌓아두는 저장소다. |

또한 3.0에서는 UI를 **React** 기반으로, 백엔드를 **FastAPI** 기반으로 전환하며 성능과 사용성을 동시에 잡으려는 시도가 엿보인다. 

![Airflow 3.0 메인 대시보드](image_9f084a.png)

실제로 마주한 Airflow 3.0의 대시보드는 이전 세대를 직접 경험해보지 않았음에도 불구하고, 시스템 상태를 한눈에 파악할 수 있을 만큼 매우 직관적으로 설계되었다는 인상을 준다. 71%의 성공률과 28%의 실패율을 즉각적으로 보여주는 가시성은 운영 효율 측면에서 큰 강점이 되지 않을까 생각한다.

## 3. TaskFlow API: 더 파이썬답게 짜는 방식

[cite_start](/image/airflow-ui.png)Airflow 2.0부터 도입되어 3.0에서 더 비중이 커진 **TaskFlow API**는 기존 방식보다 훨씬 직관적으로 DAG를 짤 수 있게 해주는 도구라고 생각한다[cite: 669, 670].

### 주목할 만한 포인트
1. **직관적인 정의:** `@dag`나 `@task` 같은 데코레이터를 활용해 일반 파이썬 함수를 태스크로 바꾸는 방식이 요즘 트렌드인 것 같다.
2. **자동 의존성 설정:** 함수를 호출하는 순서에 따라 태스크 사이의 연결 고리(Edge)가 자동으로 만들어지는 점이 매우 편리하게 느껴진다.
3. **깔끔한 데이터 전달:** 태스크 사이의 데이터 이동이 XCom을 통해 자연스럽게 처리되어 코드가 군더더기 없이 깔끔해지는 이점이 있다.

> **생각해볼 점:** DAG 파일은 실제 로직이 돌아가는 곳이라기보다 **설계 선언문**에 가깝다. 따라서 파싱 단계에서 너무 무거운 연산을 넣는 것은 피하는 것이 좋겠다고 생각한다.

## 4. 실전에서 느낀 편의성: 버전 관리와 로그 추적

실습을 진행하며 가장 만족스러웠던 부분은 코드의 수정 이력을 관리하는 방식과 문제 발생 시의 대응 과정이었다.

### 코드 버전 관리 (Versioning)
![DAG 버전 관리 화면](/image/airflow-version.png)

DAG Run을 실행 시점의 DAG 정의와 연결하여 운영 재현성을 강화했다는 점이 눈에 띈다. 실습 중 코드를 업데이트할 때마다 v1, v2로 버전이 기록되는 것을 보며, 대규모 운영 환경에서 코드 변경의 영향을 추적하기에 최적화된 구조라는 확신이 들었다. 저장 버튼 하나로 이력이 남는다는 점이 엔지니어에겐 정말 든든한 기능이 아닐까 싶다.

### 직관적인 트러블슈팅
![실패 로그 확인 화면](/image/airflow-error.png)

문제가 생겼을 때 굳이 터미널에서 도커 로그(`docker logs`)를 뒤지지 않아도 된다는 점은 혁신적이다. UI 내에서 실패한 태스크의 로그를 즉시 확인할 수 있고, `ZeroDivisionError` 같은 에러 발생 위치를 코드 라인 단위로 짚어주는 기능 덕분에 트러블슈팅 시간이 비약적으로 단축되는 것을 경험했다.

## 5. 실행 단위에 대한 고찰 (DAG Run과 Task Instance)

운영 중에 발생하는 문제를 잘 추적하려면 실행 단위를 구분해서 보는 것이 중요하다고 생각한다.

- **DAG Run:** 스케줄에 따라 혹은 수동으로 만들어진 전체 DAG의 실행 한 건을 의미한다.
- **Task Instance:** 그 한 건의 DAG 실행 안에서 개별 작업이 실제로 돌아가는 단위라 할 수 있다.
- **Execution Date (Logical Date):** 실제 시계 시간보다는 해당 작업이 처리해야 할 **데이터의 논리적 기준점**으로 이해하는 것이 적절해 보인다.

---

## 6. Docker를 활용한 운영의 효율성

Airflow 3.0은 컴포넌트가 워낙 많다 보니, **Docker**를 써서 환경을 격리하고 어디서든 똑같이 돌아가게 만드는 것이 필수적인 흐름이라 생각한다.

- **환경 재현성:** 내 로컬 컴퓨터와 클라우드(AWS EC2) 환경 사이의 간극을 줄여주는 핵심 요소다.
- **독립적 확장:** 스케줄러나 워커 등을 각각의 컨테이너로 띄워 필요할 때마다 따로 늘려줄 수 있는 점이 매력적이다.
- **의존성 관리:** 복잡한 패키지들을 컨테이너 안에 깔끔하게 가두어 관리하는 방식이 운영 안정성 면에서 유리해 보인다.

---

## 결론

Airflow 3.0은 단순히 스케줄을 관리하는 도구를 넘어, API 중심의 현대적인 아키텍처로 나아가고 있는 것 같다. **역할의 명확한 분리**를 통해 안정성을 챙기면서도, **TaskFlow API**를 통해 개발자가 파이썬답게 코딩할 수 있는 환경을 만들어준 점이 인상 깊다. 특히 실습을 통해 경험한 버전 관리와 내장 로그 뷰어는 복잡한 시스템 운영의 난이도를 낮춰주는 결정적인 요소이지 않을까 생각한다. 결국 엔지니어는 이러한 구조를 잘 이해하고, 효율적인 '설계도'를 그리는 데 집중하는 것이 정답이지 않을까 싶다.
