## 개요

Airflow를 처음 접했을 때는 DAG를 만들고 돌리는 것 자체에 집중하게 되는 것 같다. 그런데 조금 더 들어가다 보면, **언제 실행할지**, **실패하면 어떻게 할지**, **과거 데이터는 어떻게 채울지**, 그리고 **Task끼리 어떻게 데이터를 주고받을지**까지 고민이 점점 쌓이기 시작한다. 이번 포스팅은 그 고민들을 한데 모아 정리해보려 한다.

---

## 1. Schedule, Catchup, Retry — 실행 타이밍의 삼각 구도

Airflow에서 DAG의 실행 시점을 결정하는 핵심 파라미터들을 이해하는 것이 순서인 것 같다.

### cron 표현식

DAG 실행 주기를 정의하는 가장 기본적인 방법은 **cron 표현식**이다. `분 시 일 월 요일` 순서로 다섯 자리를 구성하는 방식인데, 처음에는 다소 생소하게 느껴질 수 있지만 익숙해지면 꽤 직관적이라는 인상을 받는다.

| 구분 | 표현 | 의미 |
|------|------|------|
| 기본 예시 | `0 0 * * *` | 매일 자정 |
| 간격 지정 | `0 */6 * * *` | 6시간마다 |
| 범위 지정 | `1-5` | 1부터 5까지 |
| 특정 값 | `1,3,5` | 1, 3, 5에만 |
| Airflow 단축어 | `@daily` | 매일 자정 |
| Airflow 단축어 | `@hourly` | 매시간 시작 |

> **Remind:** Airflow의 스케줄은 기본적으로 **UTC 기준**으로 동작한다. 한국 시간(KST)은 UTC+9이므로, 원하는 실행 시간에서 **9시간을 뺀 시간**으로 작성을 해야되지만, pendulum 라이브러리를 이용해 변경은 가능하다.

### Schedule과 실행 타이밍

`schedule_interval`을 설정하면 DAG가 언제 실행될지 결정되는데, 여기서 헷갈리기 쉬운 개념이 하나 있다. 바로 **logical date**와 **실제 실행 시점**의 차이다.

- `schedule_interval`: `@daily` 또는 `0 0 * * *`와 같이 실행 주기를 설정하는 값
- `data interval`: DAG가 처리해야 하는 데이터의 구간
- `logical date`: 데이터 처리 기준 날짜. 실제 실행 시점과 다르다
- `실행 시점`: `schedule_interval` 이후에 실제 실행됨 (지연 발생)

쉽게 말하면, `@daily`로 설정된 DAG는 **1월 1일 데이터**를 처리하기 위해 **1월 2일 자정에 실행**된다. logical date는 1월 1일이고, 실제 실행은 1월 2일이다. 이 차이를 모르고 운영하면 로그를 보다가 "왜 날짜가 하루 전이지?" 하는 혼란이 생길 수 있다.

### Retry - 실패 복구 메커니즘

일시적인 오류가 발생했을 때 Task를 자동으로 재시도하는 기능도 세심하게 설계되어 있다. 주요 파라미터를 정리하면 아래와 같다.

| 파라미터 | 설명 |
|----------|------|
| `retries` | 실패 시 재시도 횟수를 지정 |
| `retry_delay` | 재시도 간격을 설정 |
| `max_retry_delay` | 최대 재시도 간격을 제한 |
| `retry_exponential_backoff` | 대기 시간을 지수적으로 증가시키는 설정 (True/False) |
| `execution_timeout` | Task의 최대 실행 시간을 제한 |

운영 측면에서 Retry를 적용할 때는 문법 오류처럼 **'다시 돌려도 무조건 실패할 오류'가 아닌지 먼저 파악**하여 시스템 자원 낭비를 막아야 한다. 또한, 재실행 시 데이터가 중복 적재될 위험이 있는 **비멱등적(Non-Idempotent) Task의 경우 반드시 `retries=0`으로 예외 처리**를 해주는 것이 핵심이다.

## 2. Backfill - 양날의 검

이번 내용에서 가장 집중해서 살펴본 개념은 Airflow 운영의 **'양날의 검'**이라 불리는 **Backfill**이다.

Backfill은 한마디로 **과거 특정 기간에 대해 DAG를 실행하는 기능**이다. 단순히 실패한 Task를 다시 돌리는 Retry와는 결이 다른데, 아예 **새로운 DAG Run을 생성**해서 과거 데이터를 재처리하는 방식이다. 이런 특징 덕분에 누락된 과거 데이터를 한 번에 복구하는 매우 강력한 무기가 되지만, 통제 없이 실행할 경우 워커 자원을 순식간에 고갈시키고 중복 데이터를 만들어내는 치명적인 사고를 칠 수 있어 진정한 양날의 검이라고 할 수 있다.

### Backfill이 필요한 상황

실제 운영에서 Backfill이 필요해지는 시나리오는 생각보다 다양하다.

| 상황 | 설명 |
|------|------|
| 장애 | 시스템 다운으로 인한 데이터 처리 실패 복구 |
| 배포 오류 | 코드 버그로 인해 잘못 처리된 데이터 정정 |
| 데이터 지연 | 소스 데이터 지연으로 인한 불완전한 처리 보완 |
| 로직 변경 | 비즈니스 로직 변경으로 인한 과거 데이터 재처리 |
| 데이터 품질 | 데이터 품질 이슈 발견 후 정확한 데이터로 정정 |

### Retry vs Backfill 비교

둘의 차이를 명확히 구분해두는 것이 운영 판단에 도움이 될 것 같다.

| 구분 | Retry | Backfill |
|------|-------|----------|
| 목적 | 일시적인 오류 복구 | 과거 데이터 재처리 |
| 동작 단위 | Task 레벨 | DAG 레벨 |
| DAG Run | 동일한 DAG Run 내에서 작동 | 새로운 DAG Run을 생성 |
| 실행 범위 | 실패한 Task만 재실행 | 특정 기간의 모든 Task를 재실행 |

## 3. Catchup vs Backfill — 무엇을 선택해야 할까

비슷해 보이는 두 개념이지만 사용하는 방식과 목적이 꽤 다르다.

**Catchup**은 DAG 파라미터(`catchup=True/False`)로 제어하는 자동화 메커니즘이다. DAG가 활성화되는 시점에 `start_date`부터 현재까지 실행되지 않은 모든 간격을 스케줄러가 자동으로 채워주는 방식이다. 기본값이 `True`이기 때문에, 의도치 않게 과거 DAG Run이 한꺼번에 쏟아지는 상황을 경험할 수 있는데 이 점은 주의가 필요하다.

**Backfill**은 운영자가 CLI를 통해 명시적으로 개입하는 방식이다.

```bash
airflow dags backfill [dag_id] -s 2023-01-01 -e 2023-01-07
```

`--rerun-failed-tasks` 옵션을 붙이면 실패한 Task만 재실행하고, `--reset-dagruns`를 쓰면 기존 DAG Run을 초기화하고 재실행하는 것도 가능하다.

> **Remind:** Backfill과 일반 스케줄링이 동시에 실행될 경우 시스템 리소스를 두고 경쟁할 수 있다. `--pool`이나 `-j/--max-active-runs` 옵션을 활용해 병렬 실행 수를 제한하는 것이 안전해 보인다.

## 4. Backfill의 위험성과 중복 데이터 문제

Backfill이 강력한 도구인 만큼, 운영 환경에서는 신중하게 다뤄야 한다는 것도 중요한 내용이다. 대표적인 위험 요소를 정리하면 아래와 같다.

- **리소스 부하**: 대량의 Task가 동시에 실행되어 시스템 자원을 과도하게 사용할 수 있다
- **중복 처리**: 이미 처리된 데이터를 중복으로 처리하여 데이터가 왜곡될 가능성이 있다
- **의존성 문제**: 다른 시스템과의 의존성으로 인해 예상치 못한 부작용이 발생할 수 있다
- **데이터 정합성**: 시점 차이로 인한 데이터 불일치가 특히 외부 시스템 연동 시 문제가 될 수 있다

이를 해결하기 위한 핵심 개념이 바로 **Idempotent(멱등성) 설계**다. 같은 입력에 대해 항상 같은 결과를 보장하는 설계를 말하는데, `UPSERT`나 조건부 처리 로직이 대표적인 구현 방법이다. Backfill은 단순한 기능이 아니라 **파이프라인 설계 시점부터 고려해야 하는 설계 문제**라는 점이 인상 깊었다.

## 5. XCom — Task 간 데이터 전달

**XCom(eXchange Communication)**은 Airflow에서 Task 간에 데이터를 전달하는 기본 메커니즘이다. TaskFlow API를 사용하면 함수의 `return` 값이 자동으로 XCom에 저장되고, 다음 Task가 이를 파라미터로 받는 방식으로 동작한다.

```python
# 반환값은 자동으로 XCom에 저장됨
@task
def extract_data():
    return {'count': 100, 'status': 'success'}

# 파라미터로 자동 전달됨
@task
def process_data(data_dict):
    print(f"Count: {data_dict['count']}")
```

내부적으로는 XCom 데이터가 **Airflow의 메타데이터 DB**(SQLite/MySQL/PostgreSQL)에 저장되는 구조다. 동작 흐름을 정리하면 아래와 같다.

| 단계 | 설명 |
|------|------|
| 생성 | Task에서 값을 return |
| 저장 | 자동으로 XCom에 저장 (암시적 push) |
| 조회 | 다음 Task의 파라미터로 자동 전달 (자동 xcom_pull) |
| 활용 | 조회한 데이터를 기반으로 Task 실행 로직에 활용 |

### XCom Anti-pattern

XCom에서 가장 중요한 것은 **무엇을 전달하느냐**에 대한 판단인 것 같다.

> **XCom은 작은 메타데이터 전달에만 사용하는 것이 원칙이다.** 상태값, 카운트, 파일 경로 같은 가벼운 정보를 넘기는 용도로 설계되어 있다.

대용량 파일이나 DataFrame, ML 모델 객체를 XCom으로 전달하면 메타데이터 DB에 심각한 부하가 생기고 성능 저하로 이어진다. 대용량 데이터는 S3와 같은 외부 저장소에 저장하고, XCom으로는 **그 경로만 전달**하는 패턴이 권장 방식이라고 배웠다. 또한 오래된 XCom은 주기적으로 정리해주는 것도 운영 관점에서 중요하다는 점이 인상 깊었다.

## 6. Variable — 환경 설정값의 중앙 관리

**Variable**은 DAG 코드 안에 하드코딩되는 설정값들을 코드 바깥으로 꺼내어 중앙에서 관리하는 기능이다. Airflow의 메타데이터 DB에 저장되며, UI나 API를 통해 관리할 수 있다.

```python
from airflow.models import Variable

# 기본 사용법
bucket = Variable.get('s3_bucket_dev')

# JSON Variable
config = Variable.get('config', deserialize_json=True)

# 기본값 설정
bucket = Variable.get('s3_bucket', 'default-bucket')
```

### Variable vs Connection

비슷해 보이지만 둘의 역할은 명확히 구분된다.

| 구분 | Variable | Connection |
|------|----------|------------|
| 용도 | 환경 설정값 저장 | 외부 시스템 연결 정보 |
| 형태 | 단순 키-값 구조, JSON 지원 | 호스트, 포트, 계정 정보 |
| 보안 | 민감 정보는 암호화 | 암호화된 비밀번호 저장 |
| 활용 | 개발/운영 환경 분리 | Hook 클래스에서 활용 |

### Variable 베스트 프랙티스

운영 환경을 염두에 두면 Variable 관리에도 몇 가지 지켜야 할 원칙이 있다.

- **네이밍**: `s3_bucket_dev`, `s3_bucket_prod`처럼 환경 구분을 이름에 포함하는 것이 좋다
- **JSON 활용**: 복잡한 설정은 JSON으로 묶어서 저장하면 관리가 편하다
- **보안**: API 키나 패스워드처럼 민감한 정보는 Variable보다 AWS Secret Manager나 HashiCorp Vault와 연동하는 방향이 안전하다

## 7. AWS 연계 — Access Key 없이 인증하기

Airflow를 실제 데이터 파이프라인에 쓰다 보면 S3, Redshift, Athena 등 AWS 서비스와 연동하는 일이 자연스럽게 생기는데, 인증 방식 선택이 보안에서 꽤 중요한 부분을 차지한다는 걸 배웠다.

### IAM Role 기반 인증 — Access Key를 쓰지 않는 이유

기존에 익숙한 방식은 Access Key ID와 Secret Key를 코드에 직접 입력하는 것이다. 그런데 이 방식은 키가 코드에 노출될 위험이 있고, 주기적으로 갱신해야 하며, 관리 복잡성도 높다.

**IAM Role 기반 인증**은 이 문제를 깔끔하게 해결한다. EC2 인스턴스에 IAM Role을 부여하면, boto3가 Instance Metadata를 통해 STS(Security Token Service)에서 임시 자격증명을 자동으로 발급받는 방식이다. 코드에 키를 넣을 필요가 없고, 최소 권한 원칙도 함께 적용할 수 있다.

![인증 예시](/image/iam.png)

```python
# IAM Role 기반이면 aws_conn_id만 설정하면 된다
s3_hook = S3Hook(aws_conn_id='aws_default')
# 로그인 정보 없이 자동으로 IAM Role 인증
```

### S3 데이터 접근

S3Hook을 활용하면 파일 업로드, 다운로드, boto3 세션 활용까지 간결하게 처리할 수 있다.

| 작업 | 코드 |
|------|------|
| 업로드 | `s3_hook.load_file('local_file.csv', 'key/path.csv', 'bucket')` |
| 다운로드 | `s3_hook.download_file('key/path.csv', 'local_file.csv', 'bucket')` |
| 세션 활용 | `session = s3_hook.get_session()` |

### Redshift/Athena 연결 흐름

Redshift나 Athena 연결도 동일한 패턴으로 이루어진다. Connection ID를 생성하고, 해당 Hook 객체를 통해 쿼리를 실행한 뒤, 결과를 return해서 XCom으로 다음 Task에 전달하는 흐름이다.

```
연결 ID 생성 → hook 객체 생성 → hook.run_query(sql) → return 결과 → XCom으로 저장 → 다음 Task에서 활용
```

IAM Role을 활용하면 Variable로 환경별 연결 정보를 분리하면서도 Access Key 관리 부담 없이 안전하게 인증할 수 있다는 점이 가장 실용적인 포인트였다.

## 8. 파이프라인 설계 관점에서 바라보기

Variable과 XCom을 적절히 결합하면 파이프라인의 품질이 크게 달라진다는 것을 느꼈다. 하드코딩된 버킷명으로 직접 접근하는 기본 파이프라인과, Variable로 환경 설정을 분리하고 XCom으로 Task 간 메타데이터를 효율적으로 전달하는 개선된 파이프라인 사이에는 **코드 재사용성, 환경 분리, 유지보수 용이성** 측면에서 상당한 차이가 생긴다.

## 결론

Schedule, Catchup, Retry, Backfill, XCom, Variable, AWS 연계까지 — 이번에 다룬 내용들은 하나하나 따로 보면 개별 기능처럼 보이지만, 결국 모두 **고가용성, 안정성에 포커스한 파이프라인을 설계하기 위한 도구들**이라는 공통점이 있다. 특히 Backfill은 멱등성 설계와 세트로 이해해야 하고, XCom은 용도를 지켜서 써야 하며, AWS 인증은 IAM Role 기반으로 가져가는 것이 정답이라는 것을 이번에 제대로 정리할 수 있었다. 예외는 항상 존재하지만, 결국 데이터 엔지니어는 이 도구들의 특성을 잘 이해하고 상황에 맞게 선택하는 판단력을 키우는 것이 중요하지 않을까 싶다.
